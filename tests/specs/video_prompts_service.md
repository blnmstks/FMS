# Spec: `app/services/video_prompts.generate_video_prompts`

## Signature
```python
def generate_video_prompts(
    scenario: str,
    visual_style: dict,
    characters: list[dict],
    segments: list[dict],
    first_beat_image_path: str | None = None,
    image_prompt: dict | None = None,
) -> dict
```

## Contract

По полному сценарию, Visual Style Profile (поля строки `visual_styles`), набору персонажей
(`characters_sheet`) и **уже существующим сегментам** сценария (с вложенными битами) через LLM
генерирует для **каждого бита** видео-промпт (`video_prompt`) и описание финального кадра
(`end_frame`). Сегменты и их биты заданы извне (созданы на шаге 9, длительности посчитаны на шаге
11) — сервис их НЕ режет и НЕ выдумывает: один LLM-проход покрывает все биты всех сегментов сразу,
чтобы работала сцепка `end_frame[i] → вход клипа[i+1]`.

Вход — **сегмент-группированный**: модель видит, какие биты принадлежат одному сегменту (один
спикер, одна просодическая единица). Доктрина камеры — **«tripod-first»** (v6): по умолчанию камера
СТАТИЧНА на каждом бите, на сегмент держится ОДНА мастер-композиция (лицо в ней читаемо), движение
камеры — редкое, мотивированное сценарием, и всегда завершается оседанием в статичный кадр ВНУТРИ
одного бита (стык битов — стоп-кадр: скорость камеры не переживает handoff, бесшовно сцепляются
только статичные состояния). Смена композиции — только на границе сегмента, одиночным завершённым
move на первом бите нового сегмента.

### Inputs
| Param | Type | Описание |
|-------|------|----------|
| `scenario` | str | текст сценария идеи (`audio_beats_generated`) |
| `visual_style` | dict | поля визуального стиля клипов (значения `VISUAL_STYLE_FIELDS`) |
| `characters` | list[dict] | персонажи сценария с их данными (строки `characters_sheet`) |
| `segments` | list[dict] | заданные сегменты в порядке сценария: `{seg_id, speaker, emotion, tts_text, duration, beats: [{id, audio_text, duration}]}`; `beats` упорядочены по `id` |
| `first_beat_image_path` | str \| None | путь к РЕАЛЬНОЙ картинке шага 8 — буквальному входному кадру бита 1. Передаётся как изображение (vision) |
| `image_prompt` | dict \| None | строка `image_prompts` шага 7 — текстовое описание, по которому картинка генерировалась (поля `IMAGE_PROMPT_FIELDS`) |

User-content — текстовый блок: `GENERATE_VIDEO_PROMPTS_PROMPT` + `Scenario:` + текст
сценария + `Visual Style Profile:` (строки `ключ: значение` по `visual_style`) +
`Characters:` (`json.dumps(characters, ensure_ascii=False, indent=2, default=str)` — у
персонажей `face`/`outfit` приходят как dict из JSONB) + `Segments:`
(`json.dumps(segments, ensure_ascii=False, indent=2, default=str)` — сегменты с вложенными битами);
при переданном `image_prompt` — плюс секция `First-beat image prompt:` (строки `ключ: значение`
по непустым `IMAGE_PROMPT_FIELDS`). При переданном `first_beat_image_path` content user-сообщения
становится мультимодальным: `[{"type":"text","text":…}] + encode_images([first_beat_image_path])`
(паттерн `visual_styles`) — LLM ВИДИТ реальный стартовый кадр бита 1 и обязана начать бит 1 ровно
из него (граундинг против «телепорта» сцены: на idea-1 картинка была «в машине», промпт бита 1 —
«за столом», и первые полсекунды клипа — уродливый морф). Без картинки — прежний текстовый вызов.
System-сообщение — «You are a JSON API…». Вызывает
`create(model=DEFAULT_MODEL, messages=[system, user], response_format={"type":"json_object"})`.
Печатает usage (`prompt_tokens / completion_tokens / total_tokens`). Парсит
`json.JSONDecoder().raw_decode(response.choices[0].message.content.strip())`.

### Output
Возвращает полный распарсенный dict ответа LLM формы
`{ "beats": [ {id, video_prompt, end_frame} ] }`, где `id` — тот же `audio_beat_prompts.id`,
что был передан во входных битах (плоско, по всем сегментам, в порядке `id`). Формат выхода
**не меняется** относительно прошлой версии — по-битный. Хранение — отдельная задача
(`db.replace_video_prompts`); сервис только возвращает dict.

### Invariants
1. LLM вызывается **с** `response_format={"type":"json_object"}`.
2. В запрос попадают текст сценария, значения `visual_style`, данные персонажей и заданные сегменты
   с их группировкой битов (`seg_id`/`speaker`, а у битов — `id`/`audio_text`).
3. Изображения: при `first_beat_image_path=None` — прежний текстовый вызов (content — строка,
   обратная совместимость); при переданном пути content — список
   `[{"type":"text",…}] + encode_images([путь])`, и текст содержит секцию
   `First-beat image prompt:` (если передан `image_prompt`).
4. Невалидный JSON → пробрасывается `json.JSONDecodeError`.
5. `GENERATE_VIDEO_PROMPTS_PROMPT` несёт доктрину камеры **«tripod-first»** (v6), граундинг
   бита 1 на приложенный кадр (v7) и соответствие официальным гайдам LTX-2 (v8):
   - **Графика и текст (v8)**: LTX-2 не генерирует читаемый текст («cannot render readable or
     consistent text») — НИКОГДА не просить надписи/числа/чеки/логотипы/UI/мелкие иконки (корень
     «чека с каракулями» и каши бита 35); максимум ОДИН крупный простой текстово-свободный объект
     в негативном пространстве вдали от лица, иначе графику убрать.
   - **Аудио (v8)**: пайплайн ia2v — голос задан отдельной дорожкой, клип липсинкается под неё;
     в промпт НЕ кладутся реплики/кавычки/voiceover/музыка/SFX (это режим t2v, не наш), только
     видимая артикуляция.
   - **Детализация движения (v8)**: ~3–6 предложений ДЕТАЛЬНОГО движения «natural sequence
     flowing from beginning to end» + «camera's relationship to the subject», но ТОЛЬКО динамика;
     одно ясное действие на бит; запрет быстрых/хаотичных/прыжковых/кручёных движений и
     перегрузки.
   - **Граундинг бита 1 (v7)**: приложенное изображение — буквальный входной кадр бита 1
     («the literal input frame of beat 1»); `video_prompt` бита 1 ОБЯЗАН анимировать ровно
     изображённую сцену и кадровку — «no restaging, no teleports, no new room»; если сценарию
     нужна другая мастер-композиция — переход прописывается как ВИДИМОЕ на экране действие
     внутри бита 1, завершающееся оседанием.
   - **Static-first**: дефолт каждого бита — статичная камера, точная фраза в `video_prompt`:
     «Static camera. Locked frame. No camera movement.»; энергию кадру даёт перформанс субъекта
     (жесты, взгляд, поза, реквизит, артикуляция реплики), а не камера.
   - **Complete & settle**: движение камеры разрешено только с мотивацией из сценария, стартует из
     неподвижности, завершается и оседает в статичный кадр ДО конца бита («settles into a static
     hold»); движение НИКОГДА не пересекает границу бита («A camera move NEVER crosses a beat
     boundary») — стык битов это стоп-кадр, скорость не переживает handoff.
   - **Ритм**: не более одного move-бита на сегмент («at most ONE move-beat per segment»), никогда
     два подряд; большинство сегментов полностью статичные.
   - **Композиция**: одна мастер-композиция на сегмент («ONE master framing per segment»), лицо в
     ней читаемо; смена — только на границе сегмента, одиночным завершённым move на первом бите.
   - **I2V-минимализм**: стартовый кадр уже задан — `video_prompt` не пере-описывает видимую
     статику; идентичность — короткий identity-тег (имя + 2–3 признака из locked sheet), а не
     полная вставка Character Sheet.
   - **Сцепка `end_frame`**: финальный кадр клипа = первый кадр следующего («first frame of the
     next clip») и единственный носитель идентичности в I2V-цепочке. На статичном бите `end_frame` —
     та же неизменная мастер-композиция (без re-framing-языка); на move-бите — осевшая новая
     композиция. Object-only / «no characters are present» / обрезанное / спиной к камере ЗАПРЕЩЕНЫ
     (FORBIDDEN) на любом бите; `end_frame` не дублирует `video_prompt` («MUST DIFFER from
     video_prompt»).
   - **Непрерывность субъекта на стыке (v9)**: не только камера — поза/руки/реквизит/мимика тоже
     не переживают границу бита (стык = стоп-кадр). Доктрина несёт блок «CONTINUITY ACROSS THE BEAT
     BOUNDARY»: старт бита = состояние `end_frame` предыдущего бита (положение рук, реквизит в руках,
     поза, взгляд, выражение); действие пишется как НАЧИНАЮЩЕЕСЯ из покоя и развивающееся, НЕ как
     уже завершённое (процесс, описанный готовым состоянием, заставляет модель «прыгнуть» в него на
     frame 0 → шов). Реквизит: предмет либо уже в руке на входном кадре, либо входит в кадр по
     описанному пути — НИКОГДА не «появляется» в руке между битами; запрет метафор/намёков на
     реквизит («to suggest holding…», «as if holding…»). `end_frame` ОБЯЗАН фиксировать полное
     состояние handoff (руки/реквизит/поза/мимика), а не только читаемость лица — незафиксированная
     поза рук ровно то, из-за чего следующий клип прыгает. Мимика: выражение на старте бита =
     выражению предыдущего `end_frame`; эмоциональный сдвиг отыгрывается видимо внутри бита.
   Инвариант проверяется regression-guard'ом `tests/unit/test_video_prompts_prompt.py`
   (наличие ключевых фраз в константе).

### Test cases
- **парсит JSON в dict**: мок LLM возвращает `{"beats": [...]}` → dict содержит ключ `beats`
- **response_format задан**: `create()` вызван с `response_format={"type":"json_object"}`
- **данные в запросе**: user-content содержит текст сценария, значение поля стиля, имя персонажа,
  а также поля сегмента — `speaker` и `id`/`audio_text` вложенного бита
- **с картинкой**: при `first_beat_image_path` content — список: первый элемент — text со всеми
  данными И секцией `First-beat image prompt:` (значения `image_prompt`), далее — image_url
- **без картинки**: content — строка (старое поведение), `First-beat image prompt` отсутствует
- **битый JSON**: контент не-JSON → `pytest.raises(json.JSONDecodeError)`
